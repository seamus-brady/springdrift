## Overview

The `knowledge/` directory is Springdrift's document layer — the bridge between the outside world (operator uploads, web research, federated exchanges) and the agent's internal memory stores (narrative, CBR, facts). It holds three kinds of documents:

1. **Sources** — inbound knowledge the agent studies and absorbs
2. **Consolidation** — internal synthesis the agent produces from its own memory
3. **Exports** — outbound reports and deliverables the agent produces for humans

All three are stored as markdown files with a unified index. Documents are immutable once written — updates create new versions. The existing append-only principle extends to documents.

## Document Lifecycle

Every document follows a lifecycle tracked in `index.toml`:

### Sources (inbound)

```
Uploaded → Pending → Normalised → Studied → Promoted → (Stale on update)
```

| Status | Meaning |
|---|---|
| Pending | In inbox, not yet normalised |
| Normalised | Cleaned markdown in sources/, not yet studied |
| Studied | Ingestor agent has run a study cycle, narrative written |
| Promoted | Archivist has processed study narratives into CBR cases and facts |
| Stale | Source content changed, study cycle needs rerun |

### Consolidation (internal)

```
Generated → Available → (Superseded by next consolidation)
```

Consolidation reports don't have a complex lifecycle — they're generated by the Remembrancer and immediately available. Each new consolidation supersedes the previous one for the same period.

### Exports (outbound)

```
Draft → Reviewed → Final → (optionally) Delivered
```

| Status | Meaning |
|---|---|
| Draft | Generated by the agent, not yet quality-checked |
| Reviewed | Passed D' output gate |
| Final | Operator confirmed |
| Delivered | Sent to stakeholders via comms agent |

## Knowledge Sources (Inbound)

### Operator Upload

The operator drops files into the inbox via the web GUI Documents panel or the filesystem:

- Supported formats: markdown, plain text, PDF (text extracted), Word (text extracted)
- The Ingestor agent normalises to clean markdown
- Operator provides domain and slug (or they're inferred from filename)
- Source is immutable once normalised — updates create versioned copies

See: [Learner Ingestion spec](learner-ingestion.md)

### Agent-Discovered

The researcher agent can store web content as knowledge sources:

- `store_result` currently writes to artifacts (large content, 50KB truncation)
- A new `store_as_source` tool writes to `knowledge/sources/` instead — for content worth studying, not just referencing
- The Ingestor then runs a study cycle as normal

This closes the loop: the researcher finds something, stores it as a knowledge source, the Ingestor studies it, the Archivist promotes it to CBR cases. The agent teaches itself from its own research.

### Federated

In the federation model (see [parallel-agents-and-federation.md](parallel-agents-and-federation.md)), a federated peer can share documents:

- Received via `FederatedBroadcast` with document content
- Written to `knowledge/sources/{peer_name}/` with provenance tracking
- D' input gate evaluates before acceptance
- Study cycle runs with lower base confidence (federated source discount)

### Study Cycle Output

The Ingestor agent reads a source and produces narrative entries — not extracted records. These flow through the normal Archivist pipeline:

```
Source document
  → Ingestor reads and reasons about it
  → NarrativeEntry records written (cycle_id prefix: study:{slug})
  → Archivist processes into CbrCase (category: DomainKnowledge) + MemoryFact
  → Librarian indexes
  → Agent can retrieve via recall_cases and memory_read
```

No new retrieval mechanism. The existing CBR pipeline handles study-derived knowledge identically to experience-derived knowledge.

See: [Learner Ingestion spec](learner-ingestion.md)

## Consolidation (Internal)

The Remembrancer produces three types of consolidation documents.

See: [Remembrancer spec](remembrancer.md)

### Weekly Consolidation Reports

Synthesise a week of narrative entries into:
- Key findings and trends
- Facts restored (confidence verified)
- Patterns detected (potential skill proposals)
- Dormant threads worth revisiting
- CBR quality summary

### Pattern Reports

When the Remembrancer detects CBR clusters worth codifying:
- Pattern description with supporting case IDs
- Proposed skill text
- Confidence score and coverage metrics

These feed into the skills management pipeline.

See: [Skills Management spec](skills-management.md)

### Thread Summaries

Periodic synthesis of long-running threads:
- What the thread investigated
- Current state of knowledge
- Open questions
- Connections to other threads

## Exports (Outbound)

### Agent-Generated Reports

When the agent produces a substantial report (via the writer agent or cognitive loop), it can be saved as an export:

```gleam
pub fn save_export(
  title: String,
  content: String,
  format: ExportFormat,    // Markdown | Html
) -> Result(String, String)
```

Exports are saved to `knowledge/exports/` with metadata in the index. They appear in the Documents panel for download.

### Compliance Reports

SD Audit generates compliance reports that can be stored as exports:

```
knowledge/exports/
└── dprime-audit-2026-03.html
```

See: [SD Audit spec](sd-audit.md)

### Endeavour Deliverables

When an Endeavour completes, its final output is saved as an export:

```
knowledge/exports/
└── q2-market-analysis-final.md
```

The Endeavour's completion report links to this document. Stakeholders receive it via the comms agent.

See: [Autonomous Endeavours spec](autonomous-endeavours.md)

## Index

### Format

A single `index.toml` manifest tracks all documents across all subdirectories:

```toml
[[document]]
id = "src-legal-aamodt-plaza-1994"
type = "source"                        # source | consolidation | export
path = "sources/legal/aamodt-plaza-1994.md"
domain = "legal"
title = "Case-Based Reasoning: Foundational Issues"
status = "Promoted"
content_hash = "sha256:abc123..."
created_at = "2026-03-20T10:00:00Z"
updated_at = "2026-03-20T14:00:00Z"
source_url = "https://arxiv.org/..."
version = 1
size_bytes = 24500

[[document]]
id = "con-2026-03-23-weekly"
type = "consolidation"
path = "consolidation/2026-03-23-weekly.md"
title = "Weekly Consolidation — 2026-03-17 to 2026-03-23"
status = "Available"
created_at = "2026-03-23T09:00:00Z"
generated_by = "remembrancer"
period_from = "2026-03-17"
period_to = "2026-03-23"
size_bytes = 3200

[[document]]
id = "exp-q2-market-analysis"
type = "export"
path = "exports/q2-market-analysis-final.md"
title = "Q2 Market Analysis — Final Report"
status = "Final"
created_at = "2026-03-25T18:00:00Z"
generated_by = "writer"
endeavour_id = "end-abc123"
dprime_score = 0.12
delivered_to = ["alice@example.com"]
size_bytes = 15800
```

### Queries

The index supports:
- List by type (sources, consolidation, exports)
- Filter by domain
- Filter by status
- Filter by date range
- Full-text search (delegates to file content scanning)
- Size tracking for storage quota enforcement

## How Knowledge Flows Through the System

```
                    INBOUND
                    ═══════
    Operator upload ──→ inbox/ ──→ Ingestor normalises ──→ sources/
    Researcher finds ──→ store_as_source ──→ sources/
    Federation peer ──→ D' input gate ──→ sources/
                                              │
                                    Ingestor study cycle
                                              │
                                              ▼
                                    NarrativeEntry records
                                              │
                                    Archivist processes
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                              CbrCase              MemoryFact
                          (DomainKnowledge)    (with provenance)
                                    │                   │
                                    ▼                   ▼
                              recall_cases        memory_read
                           (agent retrieves)   (agent retrieves)

                    INTERNAL
                    ════════
    Remembrancer reads raw JSONL files
        │
        ├──→ Consolidated facts ──→ memory/facts/ ──→ agent retrieves
        ├──→ Upgraded CBR cases ──→ memory/cbr/ ──→ agent retrieves
        ├──→ Skill proposals ──→ Skills admin ──→ operator approves
        └──→ Consolidation reports ──→ knowledge/consolidation/
                                              │
                                              ▼
                                    Documents panel (operator reads)

                    OUTBOUND
                    ════════
    Agent generates report ──→ D' output gate ──→ knowledge/exports/
                                                        │
                                              ┌─────────┴─────────┐
                                              ▼                   ▼
                                    Documents panel        Comms agent
                                   (operator downloads)  (stakeholder delivery)
```

## Relationship to Memory Stores

| Store | Purpose | Knowledge Directory Role |
|---|---|---|
| Narrative | Per-cycle records | Study cycles produce narrative entries |
| CBR | Problem-solution-outcome patterns | Study cycles produce DomainKnowledge cases |
| Facts | Key-value working memory | Study cycles produce facts with provenance |
| Artifacts | Large content on disk (50KB cap) | Knowledge sources are the permanent version of what artifacts hold temporarily |
| Knowledge sources | Operator-supplied domain documents | THE input to the study pipeline |
| Knowledge consolidation | Remembrancer synthesis | THE output of the deep memory review |
| Knowledge exports | Agent deliverables | THE output for human consumption |

The key distinction: **memory stores** are for the agent's internal use (indexed in ETS, queried by tools). **Knowledge documents** are for the human-agent boundary (readable in the web GUI, downloadable, shareable).

## Provenance Chain

Every piece of knowledge tracks where it came from:

```
Source document (knowledge/sources/legal/aamodt-plaza-1994.md)
  → Study cycle narrative (memory/narrative/, cycle_id: study:legal/aamodt-plaza-1994:001)
    → CbrCase (memory/cbr/, source_narrative_id: study:legal/aamodt-plaza-1994:abc123)
      → Retrieved by agent in cycle xyz789
        → Used in report (knowledge/exports/q2-analysis.md)

Consolidation report (knowledge/consolidation/2026-03-23-weekly.md)
  → Generated by Remembrancer from narrative entries 2026-03-17 to 2026-03-23
    → Contains restored fact "dprime_reject_threshold" (source: direct observation)
    → Contains skill proposal (source: 5 CBR cases with IDs listed)
```

SD Audit can trace any claim in any export back through the chain to its original source.

See: [SD Audit spec](sd-audit.md), [Empirical Evaluation spec](empirical-evaluation.md)

## Staleness and Versioning

### Sources

Sources are immutable. When content changes:
1. New version written: `{slug}-v{n}.md`
2. Old version marked Stale in index
3. CBR cases from old version soft-deleted (redacted)
4. Re-study cycle runs on new version
5. Old version retained for audit trail

### Consolidation

Each consolidation supersedes the previous one for the same period. Old consolidations retained but marked as superseded.

### Exports

Exports are immutable once Final. Revised reports are new documents, not overwrites. The index tracks the revision chain.

See: [Learner Ingestion spec](learner-ingestion.md) for staleness handling detail

## D' Integration

| Document Type | D' Gate | What It Checks |
|---|---|---|
| Source (operator upload) | None — operator is trusted | — |
| Source (agent-discovered) | Output gate on researcher's assessment | Was the source evaluation accurate? |
| Source (federated) | Input gate | Is the inbound document safe? |
| Consolidation | None — internal synthesis | The facts/cases produced are subject to normal D' |
| Export (report) | Output gate | unsourced_claim, accuracy, certainty_overstatement, confidentiality |
| Export (delivered via comms) | Output gate + comms deterministic rules | No credentials, no internal URLs, appropriate for recipient |

## Web GUI: Documents Panel

Part of the unified layout (see [Web GUI v2 spec](web-gui-v2.md)).

### Three Sections

**Knowledge Base** (sources):
- Tree view by domain
- Status badges (Pending, Normalised, Studied, Promoted, Stale)
- Click to view rendered markdown
- Upload button / drag-and-drop to inbox

**Consolidation**:
- Reverse chronological list
- Weekly reports, pattern reports, thread summaries
- Click to read
- "Run Consolidation" button (triggers Remembrancer)

**Exports**:
- List of agent-generated reports and deliverables
- Status (Draft, Reviewed, Final, Delivered)
- Download as markdown or HTML
- Delivery history (who received it, when, via which channel)

### Document Viewer

Click any document to view:
- Rendered markdown in the main panel
- Metadata in the detail panel (provenance, status, version, size, creation date)
- For sources: study cycle status, derived CBR cases, derived facts
- For exports: D' score, delivery history, endeavour link

## Multi-Tenant

In multi-tenant mode (see [Multi-Tenant spec](multi-tenant.md)):

```
.springdrift/tenants/{tenant_id}/knowledge/
├── index.toml
├── inbox/
├── sources/
├── consolidation/
└── exports/
```

Each tenant has its own knowledge directory. Documents are fully isolated — one tenant's sources are never visible to another.

Knowledge sources uploaded by the operator are scoped to the tenant they're logged into. Federated documents carry the source tenant's ID in provenance.

## What This Enables

The knowledge directory is the human-agent boundary for documents. Operators put knowledge in (sources). The agent absorbs it (study cycles). The agent synthesises its own deep memory (consolidation). The agent produces reports (exports). Stakeholders receive them (comms delivery).

Every document has provenance. Every derived fact traces back to its source. Every export traces back to the cases and facts that informed it. SD Audit can follow the chain from a claim in a delivered report all the way back to the original uploaded document.

That's not a file system. That's a knowledge supply chain — auditable end to end.
